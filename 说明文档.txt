基于 K8s 的“自愈型” AI 算力平台 (AI Platform / SRE)

**一句话总结：** 机器冒烟前把任务捞出来，机器宕机后让任务自动复活。

**1. 为什么大厂需要这个？（痛点）**

大模型训练动辄几百张卡跑一两个月。如果中间有一张 A100 硬件故障（比如温度过高、ECC 内存错误、XID 报错），整个训练集群的通信（NCCL）就会瞬间崩溃，任务直接失败。

大厂SRE每天最头疼的就是半夜被警报叫醒去重启任务。他们极其需要一个**自动化容错**的平台。

**2. 核心架构设计：**

- **感知层 (DCGM + Exporter)：** 在每个 K8s Node 上跑一个 DaemonSet，利用 NVIDIA DCGM 实时抓取底层硬件指标。你用 Go 写的 Exporter 把这些数据喂给 Prometheus。
    
- **控制面 (K8s Custom Controller/Operator)：** 这是你的核心代码。用 `client-go` 或者 `kubebuilder` 写一个控制器，监听两个东西：Pod 的状态 和 GPU 的 Metrics。
    
- **自愈逻辑 (核心链路)：**
    
    1. **预警：** Prometheus 告警说 Node-3 的 GPU 出现 `XID 79` 错误（硬件级故障）。
    2. **隔离：** 你的 Operator 瞬间调用 K8s API，给 Node-3 打上 `Cordon`（不可调度）标记和 `Taint`（污点），防止新任务调度上去。
    3. **驱逐与复活：** Operator 杀掉 Node-3 上正在跑的训练 Pod。K8s 的 Job 控制器会在健康的节点上重新拉起这个 Pod。
    4. **断点续训：** 新的 Pod 启动时，从共享存储（NFS/Ceph）挂载最近一次的 Checkpoint 快照，继续训练。
        

**3. 面试时的“高光亮点”：**

- “我不只是做了一个面板来看温度，我开发了一个闭环的 Kubernetes Operator。它将 GPU 的底层硬件指标与 K8s 的调度系统打通，实现了算力节点的自动故障隔离和训练任务的分钟级自愈，极大降低了大规模集群运维的成本。”

**4. 你的落地步骤（怎么写代码）：**

1. 学习一下怎么用 Go 调 K8s 的 API (`client-go`)，写一个能自动给 Node 打标签的脚本。
    
2. 写一个服务，定期去读 Prometheus 里的假数据（模拟 GPU 报错）。
    
3. 一旦读到报错，触发你的 K8s 脚本，执行驱逐 Pod 的操作。
